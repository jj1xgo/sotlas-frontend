# Best Practices

> `lessons.md` の 26 項目を蒸留した高レベル原則集。
> 詳細な教訓・事例は `lessons.md` を参照。
> グローバル層（`~/.claude/best_practices.md`）収載済みの原則は再掲せず、
> 本ファイルにはプロジェクト固有の具体化のみを置く。

## 1. 【fork状態は実データで】

**何をするか**: fork/upstream の同期・マージ・push 状態やブランチの push 済みかどうかは、patch-id 比較や handover の記載を鵜呑みにせず、実データ（`git diff A B -- <path>` の内容 diff、`git fetch` + 実リモートの `git log origin/<branch>`・`git ls-remote` 照合）で確認する。特に rebase 等の履歴書き換えを挟んだブランチは記載が古くなりやすい。

**なぜ有効か**: patch-id 比較の誤判定（1コミット複数目的）と、handover の「未push」記載が実際のリモート状態と食い違っていた事例（rebase 以前の旧系譜が残存）を、どちらも実データ照合で検知できた。

**よくある失敗パターン**: コミット単位のツール出力や引き継ぎ記載を鵜呑みにして同期・push状態を断定する。

## 2. 【記述は実装・実データで裏取り】

**何をするか**: 「これはこの fork/プロジェクトだけのローカル事情」「一覧はこれで尽くしている」といったドキュメント記述は、upstream の履歴・PR（`git log --all`・`git branch --contains`）や `git ls-files`/`git log` 等の実データと定期的に突き合わせる。

**なぜ有効か**: 「ローカルパッチ」と思っていた FontAwesome 対応が実は upstream にマージ済み（PR #43）だったケースと、「master固有のみ」と判断した `.claude/` 一覧記述が実際には既に master にコミット済みの調査ノートを見落としていたケースの両方を、実データ照合で訂正できた。

**よくある失敗パターン**: CLAUDE.md 等の記述だけを根拠に現状（ローカル事情か否か・一覧の網羅性）を判断する。

## 3. 【動作検証は実行経路全体で】

**何をするか**: 「このライブラリは Vue 3 / Vite で動くか」の検証は、Node 単体の require/import・`renderToString` による SSR だけで完了とせず、`npm run build`・実ブラウザ（`app.mount()` まで）を通す。Node/SSR のモジュール解決やライフサイクル実行は実バンドラ・実クライアントと異なりうる。

**なぜ有効か**: Node 単体検証は CJS/ESM 解決の違いで `install()` 本体に到達しない偽陰性を生み、`renderToString` は `mounted()` 等を実行しないため子コンポーネントの自己登録設計を偽陰性のまま見逃した。実経路まで通す検証を完了条件にしてどちらも捕捉できるようになった。

**よくある失敗パターン**: Node/SSR でのスクリプトが例外を投げなかったことを「実ブラウザでも動く」と読む。

## 4. 【描画の暗黙デフォルトを明示】

**何をするか**: 描画ライブラリの省略時デフォルト（例: MapLibre の `fill` レイヤーは `fill-outline-color` 未指定だと `fill-color` と同色でアウトラインを描く）が絡む見た目のアーティファクトは、明示指定（透明化等）で潰す。半透明ポリゴンの重ね合わせでは特に注意する。

**なぜ有効か**: 縮退辺を持つポリゴンで自動アウトラインが重なり線状アーティファクトになった。明示的な透明化で解消し、以後の重ね描画実装の定石になった。

**よくある失敗パターン**: アーティファクトの原因を自分のジオメトリ生成側だけに探し、レンダラの暗黙仕様を疑わない。

## 5. 【許可リストは実走で確定】

**何をするか**: allowed-domains 等の環境許可リストは、`package.json` の依存を全種類（registry / `github:` / `git+https:` 等）洗い出して設計し、実際にコンテナ内で `npm ci` が通るまで完成としない。

**なぜ有効か**: `github:` 形式の依存が `codeload.github.com` から tarball を取ることを、registry.npmjs.org だけ許可して実走するまで見落としていた。実走確認を完成条件にして検知できた。

**よくある失敗パターン**: 依存の取得経路を registry 一種と仮定して許可リストを机上で確定する。

## 6. 【メジャーバージョン移行は契約差分まで確認】

**何をするか**: ライブラリのメジャーバージョン間互換性は、props/emits 等の名前一致だけで判断せず、呼び出し契約（スロットの渡し方等）と、Vue 3 のようなフレームワーク側のコンパイル規則変更（named slot と並ぶ v-slot なしの裸 `<template>` が暗黙 default として扱われず宣言のまま消える等）を公式サンプル・コンパイラ出力で照合する。

**なぜ有効か**: Buefy 0.8→3.0 で props/emits 名は不変だったがスロット呼び出し契約が変わっており、ビルドは通り見た目もヘッダーまで出るため気づきにくいバグを、公式サンプルとの照合で検知できた。裸 template の inert 化も同様にエラーが出ず気づきにくい。

**よくある失敗パターン**: props/emits 名が一致していることを互換性ありの証拠として扱う。

## 7. 【git操作の副作用を先に把握】

**何をするか**: コミット直後の `git checkout` によるブランチ切替は「現在 HEAD で tracked だが切替先に存在しないファイル」を黙って削除する。また `git add` は複数パススペックのうち1つでも一致しないと全体を失敗させる。これらの副作用を踏まえ、切替直後は `git status` で状態を確認してから次の操作をする。

**なぜ有効か**: 別ブランチへのコミット直後に元ブランチへ切り替えた際、事前に計画していた手動 `rm` が不要（自動削除済み）だと `git status` 確認で気づけた。既に `git rm` 済みのパスを他の `git add` と混在させて失敗するミスも防げるようになった。

**よくある失敗パターン**: git 操作の副作用を知らずに、切替後・add の前後で想定通りの状態になっていると決め打ちする。

## 8. 【運用手順はskill化してtrigger明記】

**何をするか**: たまにしか使わない手順を CLAUDE.md の外（プロジェクト skill `.claude/skills/<name>/SKILL.md`）に出すときは、frontmatter の `description`（+`when_to_use`）に呼び出されるべき具体的な trigger 文言を書き込む。

**なぜ有効か**: 「CLAUDE.md の外に出すと呼ばれない限り読まれなくなる」という懸念は、skill の description が常時コンテキストの一覧に載り続ける仕組み（公式ドキュメント確認済み）により、trigger 文言の明記だけで実質的に解消できた。

**よくある失敗パターン**: 手順を skill へ切り出す際、description を一般的な要約のまま済ませ、呼び出されるべき具体的な状況を書かない。

## 9. 【ソースは配布物よりビルド前を優先】

**何をするか**: `node_modules/<pkg>/src/`（ビルド前ソース、TypeScript の場合あり）が存在する場合、`dist/` の minify されたコードより先にそちらを読む。

**なぜ有効か**: 新規導入ライブラリの `lib/*.ts` を直接読むことで、計画時点では未確定だったイベントペイロード形状・emits 一覧・props の仕様を実装前に正確に確定でき、原因特定・API 確認が大幅に速くなった。

**よくある失敗パターン**: `dist/` の minify されたコードや README の断片的な記述だけで API を推測する。

## 10. 【設計判断はplanに固定し逸脱時は確認】

**何をするか**: 計画ファイルに明記された設計判断（採用/却下の理由込み）を実装セッションで覆す場合、無言で進めずユーザーに確認するか、少なくとも覆した根拠を handover に明示する。

**なぜ有効か**: 前回セッションが明示的に却下した設計を、後続セッションが「公式ドキュメントと矛盾する」という理由で無断採用し直した事例があった。セッションをまたぐと計画上の決定が引き継がれず再考されるリスクを、確認・根拠明示のルール化で抑えられる。

**よくある失敗パターン**: 公式ドキュメント等のより強い根拠が見つかったからと、計画済みの設計判断を無言で上書きする。
