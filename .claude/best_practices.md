# Best Practices

> `lessons.md` の 66 項目を蒸留した高レベル原則集。
> 詳細な教訓・事例は `lessons.md` を参照。
> グローバル層（`~/.claude/best_practices.md`）収載済みの原則は再掲せず、
> 本ファイルにはプロジェクト固有の具体化のみを置く。

## 1. 【fork 状態は実データで】

**何をするか**: fork/upstream の同期・マージ・push 状態は、patch-id 比較や handover の記載でなく、実データ（`git diff A B -- <path>` の内容 diff、`git fetch` + 実リモート照合）で確認する。特に rebase 等の履歴書き換えを挟んだブランチは記載が古くなりやすい。

**なぜ有効か**: patch-id 比較の誤判定（1コミット複数目的）と handover の「未push」記載の食い違い（旧系譜がリモートに残存）を、実データ照合でどちらも検知できた。

**よくある失敗パターン**: コミット単位のツール出力や引き継ぎ記載を鵜呑みにして同期状態を断定する。

## 2. 【upstream・文書は実装と突き合わせ】

**何をするか**: 「これはこの fork だけのローカル事情」と断定する前に upstream の履歴・PR（`git log --all`、`git branch --contains`）を確認する。CLAUDE.md の「ローカルパッチは現在〇〇のみ」等の現状一覧記述も鵜呑みにせず、`git ls-files <branch>`/`git log` で裏取りしてから判断する。

**なぜ有効か**: 「ローカルパッチ」と思っていた FontAwesome 対応が実は upstream マージ済み（PR #43）だった件、「master に置けない」と判断した調査ノートが実は master にコミット済みだった件を、実データ裏取りで正せた。

**よくある失敗パターン**: CLAUDE.md 等の記述だけを根拠に実装・ブランチ内容の現状を説明する。

## 3. 【異常疑いは handover から】

**何をするか**: 想定外のコミット・未コミット差分を見つけたら、確認の順序を固定する: (1) 直近の `.claude/handovers/` 最新ファイル → (2) 並行セッションの有無をユーザーに確認 → (3) それでも説明がつかない場合のみインシデント記録。

**なぜ有効か**: 前セッションが handover に明記済みの意図的パッチを、handover を読まずに異常扱いしてインシデント記録まで進めた誤検知が、この順序の固定で防げるようになった。

**よくある失敗パターン**: 「ユーザーに聞く」や隔離フローを handover の確認より先に実行する。

## 4. 【PR 経路は実マージ履歴で】

**何をするか**: Phase 分割 PR のベースブランチ・rebase/cherry-pick の起点・PR 作成 URL は、一般則の無条件適用でなく実データで決める。ベースは `git log --oneline upstream/<branch>` の実マージ状況、起点は対象ブランチの `git log --stat`（ローカル専用コミット混在の有無）、PR 作成は `git push` が出力する fork 基準 URL + base の明示切替。

**なぜ有効か**: 後続 Phase の PR ベースが `master` でなく upstream の `vue3` ブランチだった件、作業ブランチに `.claude/` docs コミットが混在し単純 `rebase --onto` では PR にローカルパッチが混入する件、upstream 側 `pull/new` URL が fork ブランチを解決できない件を、実データ確認で回避できた。

**よくある失敗パターン**: CLAUDE.md の一般則（`git rebase --onto upstream/master master feat/<slug>`）を対象ブランチの中身を見ずに適用する。

## 5. 【git は一手ごとに実物確認】

**何をするか**: コミット直後のブランチ切替・複数パススペックの `git add` 等、複合的な git 操作は各操作後に `git status`/`git show --stat` で実状態を確認してから次の操作へ進む。`git rm` 済みのパスを他の `git add` と混在させない。ある機能追加コミットが単一ブランチにのみ存在する場合は、そのコミットが影響する全ファイル（`git show --stat`）を確認してから移植・比較する（一部ファイルだけ見て「動くはず」と誤認しない）。

**なぜ有効か**: `git checkout` が切替先に存在しない tracked ファイルを黙って削除済みで、計画していた手動 `rm` が不要だった。また `git add` は1つの不一致パスで正当なパスも含め全体が失敗する。単一ブランチ限定コミットの全ファイル確認漏れで、関連ファイル（`packages.txt` 等）のコピー漏れにも気づけた。操作後の状態確認でいずれも無駄手・誤報告を防げた。

**よくある失敗パターン**: 事前に立てた手順を、各操作後の状態を見ずに機械的に消化する。コミットの影響範囲を diff の一部だけで判断する。

## 6. 【検証は実行経路を揃える】

**何をするか**: 「Vite + 実ブラウザで動くか」の検証を Node 単体の `import`/`require` や SSR（`renderToString`）で代替しない。CJS/ESM 解決の差（`exports` フィールド無しだと Node は `main` を解決、Vite は `module` を見る）、ライフサイクルフック不実行（SSR は `mounted()` を呼ばない）が偽陰性/偽陽性を生む。jsdom + 実 `createApp().mount()`、`npm run build`、ホスト側 dev サーバまで通して確認する。

**なぜ有効か**: Node 単体検証で「動く」と判定したライブラリが実ブラウザで白画面（全ページ描画停止）になった。実バンドラ経路まで通す検証に切り替えて偽陰性/偽陽性を排除できるようになった。

**よくある失敗パターン**: Node で `import` して例外が出ないことを「Vue 3 で動く」の証明とする。

## 7. 【暗黙デフォルトは明示で潰す】

**何をするか**: レンダラ・コンパイラの省略時挙動はエラーなしで見た目だけを壊す。MapLibre の `fill` レイヤーは `fill-outline-color` 未指定だと `fill-color` 同色のアウトラインを描く（縮退辺で線状アーティファクト化）ため明示的に透明化する。Vue 3 では v-slot なしの裸 `<template>` が inert な DOM 要素になるため `v-slot:default` を明示する。

**なぜ有効か**: どちらもコンソールエラーが出ず「中身が見た目上消える/線が出る」だけの症状で、暗黙仕様の明示指定により解消・再発防止できた。

**よくある失敗パターン**: アーティファクトや消失の原因を自分のコード側だけに探し、レンダラ・コンパイラの暗黙仕様を疑わない。

## 8. 【ラッパーは移植元契約まで写す】

**何をするか**: ライブラリの自前ラッパー層を書くときは、参照元ライブラリの初期化・teardown 契約まで移植元ソース（GitHub raw 等で再取得可）と突き合わせる。map への `addControl` は「map 生成直後（load 前）に解決済みの参照」（`mapSymbol` の `ShallowRef` 等）経由で行い、dispose では移植元が null 化していない参照を独自に undefined 化しない。

**なぜ有効か**: 独自に足した `map.value = undefined` が子コンポーネントの暗黙契約を壊して unmount 時 TypeError を起こし、また load 後にしか解決しない参照経由の `addControl` が mapbox-gl-draw のポーリングフォールバックを踏んだ。移植元の契約との突き合わせで真因に到達できた。

**よくある失敗パターン**: ラッパーの公開 API だけ揃えて、初期化タイミング・dispose 時の参照保持方針を独自判断で変える。

## 9. 【無応答も失敗として設計】

**何をするか**: ネットワーク呼び出しは `.catch()` だけでは「応答が一切返らない」ケースを救えない（`.then()` も `.catch()` も呼ばれず永久 pending）。UI のローディング状態を確実に解除するには `timeout` オプションの設定を必須とする。

**なぜ有効か**: axios の true network hang でローディングが解除されない経路を、timeout 必須化で塞げた。

**よくある失敗パターン**: 「エラーハンドリングは catch で網羅した」と考え、無応答ケースを失敗系に含めない。

## 10. 【許可リストは実走で確定】

**何をするか**: allowed-domains 等の環境許可リストは、`package.json` の依存を全種類（registry / `github:` / `git+https:` 等）洗い出して設計し、コンテナ内で `npm ci` と実ページ描画まで通るまで完成としない。実行時にしか現れる外部ドメイン（ウィジェットが叩く API 等）は実走でしか検知できない。

**なぜ有効か**: `github:` 依存が `codeload.github.com` から tarball を取ることを実走まで見落とし、また Wikipedia ウィジェットの `en.wikipedia.org` 漏れは実ページ描画（トースト表示）で初めて発覚した。実走確認を完成条件にして検知できた。

**よくある失敗パターン**: 依存の取得経路を registry 一種と仮定し、机上で許可リストを確定する。

## 11. 【検証手段は計画段階で棚卸し】

**何をするか**: 検証計画を立てる前に、対象ブランチ・環境に検証手段が実在し何をカバーできるかを確認する: 切替先ブランチの `.mcp.json` の有無、Claude in Chrome の制約（OS ネイティブダイアログ・DL 履歴は不可）、コンテナ内ヘッドレスの範囲（Turnstile ブロックで `/map` 系は不可、非 map ページは完結可）。hook のロジックは該当コード片を bash で手動再現すれば同一セッション内で実物検証できる。

**なぜ有効か**: 「master から新規ブランチ」と決めた後に切替先に `.mcp.json` が無いと気づいた手戻りがあり、逆にコンテナ内検証の可否範囲を実測してからは検証の役割分担（コンテナ内/ホスト側/手動）を計画段階で確定できるようになった。

**よくある失敗パターン**: 検証手段が今の環境と同じ状態で存在する前提で計画を立て、実施段階で手段が無いことに気づく。

## 12. 【クラッシュは所有権から疑う】

**何をするか**: コンテナ内の低レベルクラッシュ（SIGTRAP/SIGSEGV 等）を見て真っ先にサンドボックス制約（capability/seccomp）を疑わず、先に書込権限・ディレクトリ所有者（`ls -ld`）等の基本的な原因を潰す。

**なぜ有効か**: headless chromium の SIGTRAP は seccomp でなく `/home/node/.config` の root 所有（crashpad の DB パス解決失敗）が原因で、`XDG_CONFIG_HOME` の変更だけで解消した。所有権確認を先行させる順序で切り分けが速くなった。

**よくある失敗パターン**: クラッシュシグナルの低レベルさからサンドボックス起因と即断し、権限変更の検討に時間を使う。

## 13. 【連携情報は実データで再確認】

**何をするか**: 「〇〇から issue が来てる」は「〇〇のリポジトリを見る」と早合点せず、まず自リポジトリの issue 一覧に〇〇起票のものが無いか確認する。session-start hook が注入する issue スナップショットは取得時点のものなので `gh issue view` で現況を再確認する。外部スレッドへの投稿文面は、対象スレッドの実文面（`gh issue view`）に実在する用語だけを使い、内部計画ファイルの呼称を持ち込まない。

**なぜ有効か**: 起票主体と起票先の取り違え、クローズ済み issue への対応着手、内部呼称（Phase 3b 等）の外部流出を、いずれも実データ確認で防げた。

**よくある失敗パターン**: 注入情報・指示文の字面だけで対応先・対応状況・用語を決める。

## 14. 【データ探索は API 直クエリ】

**何をするか**: 特定条件のテストデータ（写真なし・ルートありの summit 等）を探すときは、ブラウザの目視探索でなく、バックエンド API（`/api/summits/<code>` の `photos` 配列、GPX エンドポイント等）を直接クエリして機械的に絞り込む。

**なぜ有効か**: ブラウザ探索で難航した稀なデータ組み合わせの特定が、API 直クエリへの切り替えで効率的に完了した。

**よくある失敗パターン**: UI 上の探索に固執し、同じ条件をバックエンドに直接問い合わせられることを忘れる。

## 15. 【キー・トークンは実効範囲で】

**何をするか**: API キーの保護方式はクライアント種別で実効性が変わる: ブラウザアプリに「Allowed user-agent header」は無効（ブラウザは UA を操作できない）で「Allowed HTTP origins」を使う。コンテナの既定 `gh`（プライマリトークン）は本リポジトリ限定スコープのため、クロスリポジトリ操作は `GH_TOKEN=$(cat .../gh-token-secondary)` を都度明示指定する。MapTiler の「Allowed HTTP origins」はプロトコル・ポート番号を含めないドメイン名のみの形式（`http://localhost:5173` は `Invalid origin restriction` で弾かれる）で、`localhost` 等の hole な origin を許可しても「他者の本番ドメインでの無断使用」は防げるが「誰かが自分の PC でキーを使う」ことまでは防げない（保護の本質はキー値自体の非公開性）。upstream 所有のカスタムクラウドスタイルは開発用キーで 404 になっても MapLibre GL が標準スタイルへ自動フォールバックするため、404 だけで致命的エラーと即断しない。

**なぜ有効か**: 設定名からの直感でなく実効性で方式・トークンを選ぶことで、機能しない保護設定と GraphQL エラーによる手戻りを避けられるようになった。origin 制限の形式・保護範囲を公式ドキュメントで確認したことで `Invalid origin restriction` エラーの原因特定と、保護効果の過大評価の両方を防げた。

**よくある失敗パターン**: 設定項目名・トークンの存在だけを見て、対象クライアント・対象リポジトリで実際に効くかを確認しない。origin 制限を設定すれば個人利用まで防げると誤解する。

## 16. 【コンテナ変数は素通し保証なしで検証】

**何をするか**: claude-container 組み込みの `_FILE` 系機構（`GH_TOKEN_FILE`/`GH_TOKEN_SECONDARY_FILE` 等）は既知変数のみをハードコード処理する設計であり、独自追加変数（`VITE_MAPTILER_DEV_KEY` 等、`_FILE` 有無・direct value 形式を問わず）には自動 export・素通し・リビルド不要などの保証は及ばない。独自変数を使うときは実環境変数・ファイル実在を個別に確認し、コンテナへ渡す場合は dev server 起動と同一 shell コマンド内で明示 export する（`Bash(env:*)` の deny ルール下では `env VAR=val cmd` は使えないため shell 組み込み `export` で代替）。`/workspace` はホストと双方向マウントのため、コンテナ内で書いたファイル（`.env.local` 等 Vite が自動で読むもの）は即座にホスト側にも実在する点も踏まえて設計する。

**なぜ有効か**: `GH_TOKEN_SECONDARY_FILE` の自動 export 誤認、`VITE_MAPTILER_DEV_KEY` の素通し失敗、`.claude-container.d/env` の direct value 形式でも素通しされない誤診断を、実データ確認（`env`・ファイル実在）で正せた。セカンダリトークンは明示指定で実際にクロスリポジトリ投稿が機能することも実地確認できた。

**よくある失敗パターン**: 組み込み機構の挙動（自動 export・リビルド不要）を独自追加変数にも及ぶと類推適用する。

## 17. 【依頼前に前提の不確実性を明示】

**何をするか**: 検証されていない技術的前提に基づいてユーザーに準備作業（キー発行・ファイル設置等）を依頼するときは、その都度不確実性をその会話で明示する（計画書に書いただけでは足りない）。前提が外れて依頼が無駄になった場合は、中立的な選択肢提示だけで済ませず、まず自分の見立て違いを認める。

**なぜ有効か**: `_FILE` 機構の汎用性という未検証前提で MapTiler キー発行等をユーザーに依頼し、外れた後も中立的な言い方に終始したためユーザーから「冷たくないか」と指摘された。以後、留保の明示と見立て違いの承認をセットにする運用に改めた。

**よくある失敗パターン**: 計画書に書いた留保を実際の依頼時の会話で伝え忘れる。前提が外れた後、選択肢の提示だけで済ませ自分の見立て違いを認めない。
