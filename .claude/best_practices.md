## 1. 【fork状態は実データで】

**何をするか**: fork/upstream の同期・マージ・push 状態、ブランチ間の内容差（検証環境設定の有無・コミットの影響ファイル全量）は、patch-id 比較や handover の記載を鵜呑みにせず、実データ（`git diff A B -- <path>` の内容 diff、`git fetch` + 実リモートの `git log origin/<branch>`・`git ls-remote`、`git show --stat`）で確認する。upstream PR のマージ状態は handover の「レビュー待ち」等の記載でなく `gh pr view --json state` で機械的に確認する。コミット系譜が別のブランチ同士（PR 提出用に切り出したブランチ等）でも、`git diff <branch1> <branch2> --stat` の内容差分で「実質同期しているか」を機械的に判定でき、ハッシュ不一致だけで「未同期」と早合点しない。「既知の制約」と判断する前に、それを解消する対応が別ブランチで既に行われていないか `git diff master <branch> -- <file>`・`git log --all` で確認する。ブランチ切替を伴う計画では、切替先に検証手段（`.mcp.json` 等）が実在するかも計画段階で確認する。

**なぜ有効か**: patch-id 比較の誤判定（1コミット複数目的）と、handover の「未push」記載が実リモート状態と食い違っていた事例（rebase 以前の旧系譜が残存）を実データ照合で検知できた。単一ブランチにのみ存在する機能コミットの一部ファイルだけをコピーして「動くはず」と誤認するリスクも `git show --stat` で塞げた。`master` でマージ・issue クローズ済みの Turnstile dev バイパスが `feat/vue3-migration` に未反映だった事例も、`git diff master feat/vue3-migration -- <file>` で「既知の制約」という早合点を訂正できた。handover の「レビュー待ち」記載を `gh pr view` で再確認し、クリーンアップ・次フェーズ着手判断のタイミングを正しく取れた。

**よくある失敗パターン**: コミット単位のツール出力や引き継ぎ記載を鵜呑みにして同期・push・マージ状態を断定する。コミットハッシュの一致不一致だけで同期を判定する。別ブランチでの解決済み対応を確認せず「既知の制約」と片付ける。ブランチが違っても検証環境は同じだと決め打つ。

## 2. 【記述は実装・実データで裏取り】

**何をするか**: 「これはこの fork/プロジェクトだけのローカル事情」「一覧はこれで尽くしている」「これは既知の残課題」といったドキュメント・lessons の記述は、upstream の履歴・PR（`git log --all`・`git branch --contains`）や `git ls-files`/`git log`・`package.json`/`node_modules` の実データと定期的に突き合わせる。lessons.md の記録は記録時点のスナップショットであり、その後の実装進展で陳腐化しうるため「既知の残課題」として扱う前に都度実データで現状を再確認する。グローバル層とプロジェクト層の両方に同種の運用規約（署名ルール等）が書かれている場合は記述同士の矛盾も突き合わせる。

**なぜ有効か**: 「ローカルパッチ」と思っていた FontAwesome 対応が実は upstream マージ済み（PR #43）だったケース、「master固有のみ」の `.claude/` 一覧記述が調査ノートを見落としていたケース、lessons に「既知未修正」と記録していた vuedraggable v2 起因バグが実は `^4.1.0` へ更新済みで再現しなくなっていたケースを、いずれも実データ照合で訂正できた。署名ルールの層間矛盾も定期レビュー（claude-md-panel）で解消できた。

**よくある失敗パターン**: CLAUDE.md・lessons.md 等の記述だけを根拠に現状（ローカル事情か否か・一覧の網羅性・残課題の未解決性・規約の一貫性）を判断する。

## 3. 【動作検証は実行経路全体で】

**何をするか**: 「このライブラリは Vue 3 / Vite で動くか」の検証は、Node 単体の require/import・`renderToString` による SSR だけで完了とせず、`npm run build`・実ブラウザ（`app.mount()` まで）を通す。Node/SSR のモジュール解決やライフサイクル実行は実バンドラ・実クライアントと異なりうる。Turnstile がヘッドレスをブロックして地図が描画されない環境でも、地図が絡まないページ（非map summit詳細等）はコンテナ内ヘッドレス検証で完結でき、地図確認のみホスト側実ブラウザへ回すという役割分担で検証範囲を最大化する。

**なぜ有効か**: Node 単体検証は CJS/ESM 解決の違いで `install()` 本体に到達しない偽陰性を生み、`renderToString` は `mounted()` 等を実行しないため子コンポーネントの自己登録設計を偽陰性のまま見逃した。実経路まで通す検証を完了条件にしてどちらも捕捉できるようになった。地図非依存ページのヘッドレス完結で、Turnstile 制約下でも検証をホスト側に丸投げせず済んだ。

**よくある失敗パターン**: Node/SSR でのスクリプトが例外を投げなかったことを「実ブラウザでも動く」と読む。地図が出ないだけで全ページの検証を諦めてホスト側へ丸投げする。

## 4. 【描画の暗黙デフォルトを明示】

**何をするか**: 描画ライブラリの省略時デフォルト（例: MapLibre の `fill` レイヤーは `fill-outline-color` 未指定だと `fill-color` と同色でアウトラインを描く）が絡む見た目のアーティファクトは、明示指定（透明化等）で潰す。半透明ポリゴンの重ね合わせでは特に注意する。

**なぜ有効か**: 縮退辺を持つポリゴンで自動アウトラインが重なり線状アーティファクトになった。明示的な透明化で解消し、以後の重ね描画実装の定石になった。

**よくある失敗パターン**: アーティファクトの原因を自分のジオメトリ生成側だけに探し、レンダラの暗黙仕様を疑わない。

## 5. 【許可リストは実走で確定】

**何をするか**: allowed-domains 等の環境許可リストは、`package.json` の依存を全種類（registry / `github:` / `git+https:` 等）洗い出して設計し、コンテナ内で `npm ci` が通り、対象機能（外部 API を叩くウィジェット等）を実際に動かすまで完成としない。

**なぜ有効か**: `github:` 形式の依存が `codeload.github.com` から tarball を取ることを、registry.npmjs.org だけ許可して実走するまで見落としていた。実行時にも Wikipedia ウィジェットの通信先（`en.wikipedia.org`）が未収載でエラートーストが出る漏れが見つかり、実行時通信まで実走確認の対象に含めるようになった。

**よくある失敗パターン**: 依存の取得経路を registry 一種と仮定し、ビルドが通った時点で許可リストを完成とみなす。

## 7. 【git操作の副作用を先に把握】

**何をするか**: コミット直後の `git checkout` によるブランチ切替は「現在 HEAD で tracked だが切替先に存在しないファイル」を黙って削除する。また `git add` は複数パススペックのうち1つでも一致しないと全体を失敗させる。これらの副作用を踏まえ、切替直後は `git status` で状態を確認してから次の操作をする。

**なぜ有効か**: 別ブランチへのコミット直後に元ブランチへ切り替えた際、事前に計画していた手動 `rm` が不要（自動削除済み）だと `git status` 確認で気づけた。既に `git rm` 済みのパスを他の `git add` と混在させて失敗するミスも防げるようになった。

**よくある失敗パターン**: git 操作の副作用を知らずに、切替後・add の前後で想定通りの状態になっていると決め打ちする。

## 8. 【運用手順はskill化してtrigger明記】

**何をするか**: たまにしか使わない手順を CLAUDE.md の外（プロジェクト skill `.claude/skills/<name>/SKILL.md`）に出すときは、frontmatter の `description`（+`when_to_use`）に呼び出されるべき具体的な trigger 文言を書き込む。

**なぜ有効か**: 「CLAUDE.md の外に出すと呼ばれない限り読まれなくなる」という懸念は、skill の description が常時コンテキストの一覧に載り続ける仕組み（公式ドキュメント確認済み）により、trigger 文言の明記だけで実質的に解消できた。

**よくある失敗パターン**: 手順を skill へ切り出す際、description を一般的な要約のまま済ませ、呼び出されるべき具体的な状況を書かない。

## 9. 【型定義は読み、公開可否は実行で確認】

**何をするか**: 配布パッケージの公開型定義（`.d.ts`）とその docstring は、ライブラリ挙動や ToS/仕様グレーの評価の一次情報として先に読む。ただし `export declare` の存在を runtime export の証拠とせず、「公開 API か」は実際の import テストの実行まで裏取りする。

**なぜ有効か**: `@maptiler/sdk` のセッション課金機構が公開 doc コメントに明記されていると確認でき、「隠された内部実装のリバースエンジニアリング」という当初評価を「公言された動作の再現」へ覆せた。同時に `MAPTILER_SESSION_ID` は型定義上 `export declare` でも実際は未 export と import テストで確定できた。

**よくある失敗パターン**: 型定義の存在だけで「公開 API・利用可」と断定する。docstring を読まずに内部実装の推測でグレー評価を下す。

## 10. 【mapコントロールはload前参照で登録】

**何をするか**: MapLibre 系コントロール（mapbox-gl-draw 等）の `addControl` は、「map の `load` 後にしか解決しない reactive な値」経由でなく、ラッパーライブラリが提供する「map 生成直後（load 前）に解決済みの参照」（`@indoorequal/vue-maplibre-gl` の `mapSymbol` `ShallowRef` 等）経由で行う。

**なぜ有効か**: mapbox-gl-draw の `onAdd` は load 済み map に対して `map.on('load', ...)`（既発火済みなら二度と発火しない）＋16ms ポーリングへフォールバックする。load 前参照で登録するようにしたら、ポーリング頼みの不安定な初期化を避けられた。

**よくある失敗パターン**: 「map が手に入ってから addControl すれば安全」と考え、load 後解決の参照経由で初期化経路を不安定にする。

## 11. 【通信はtimeoutで打ち切る】

**何をするか**: UI のローディング状態を伴う HTTP リクエスト（axios 等）には `timeout` オプションを必ず設定する。`.catch()` だけでは応答が一切返らないネットワークハングを救えない。

**なぜ有効か**: `.then()` も `.catch()` も呼ばれず永久 pending になるケースがあると分かり、timeout を必須にしたらローディング解除漏れを設計段階で潰せるようになった。

**よくある失敗パターン**: 「エラーは `.catch()` で拾っているから大丈夫」と、無応答ハングのケースを考慮しない。

## 12. 【ブラウザ検証は発火・インスタンス経由で裏取る】

**何をするか**: コンテナ内の自動検証（Playwright MCP / Claude in Chrome）で不可解な挙動や「描画されない」現象を観測したら、表層の DOM・見た目だけで判定しない。(a) 合成イベント（地図ドラッグ等）は実ハンドラにコミットされないことがあるため、コード側該当パス（watch・イベントリスナー）へデバッグログを仕込んで発火有無を確認し、未発火ならユーザーの実手動操作での再現有無を確認してから「バグ」か「テスト手法のアーティファクト」かを確定する。(b) data 値は更新されているのに DOM 要素が生成されない現象は、DOM 要素から `__vueParentComponent` を辿ってコンポーネントインスタンス（`inst.proxy`/`inst.refs`）へ直接アクセスし、状態を読む・メソッドを直接呼んで「状態が更新されていない」のか「描画契約が壊れている」のかを切り分ける（ログイン等で UI から辿れないコンポーネントの強制操作にも応用可）。検証に必要な稀なデータ組み合わせは、ブラウザ目視探索でなくバックエンド API 直接クエリで機械的に絞り込む。OS ネイティブダイアログ等は自動操作では確認できない制約も踏まえる。

**なぜ有効か**: MapLibre の `dragPan` が合成マウスイベントでは内部カメラ状態を変えない現象を、切り分け（3ラウンド全てでログ発火なし＋手動操作で非再現）でテスト手法側のアーティファクトと確定でき、無用な修正を避けられた。`<b-modal>` の prop 契約バグ調査では `__vueParentComponent` 経由で「状態は正しく描画契約が壊れている」と即座に切り分けられた。「写真なし・ルートあり」の稀な Summit ページも API 直接クエリで特定できた。

**よくある失敗パターン**: 自動操作ツール経由の観測をそのまま実ユーザー環境の挙動として扱いバグ修正を始める。DOM に出ないことだけを見て状態更新ロジックばかり疑い回り道する。

## 13. 【hook注入・issue情報は実物と自リポジトリで裏取る】

**何をするか**: コマンド文字列を正規表現で grep する設計の hook は heredoc 内リテラルまで実行コマンドと誤認しうる。session-start hook が自動注入する issue スナップショット等は取得時点のもので以後の変化を反映しない。hook からの警告・注入情報は鵜呑みにせず実データ（`git status`/`git log`・`gh issue view`）で裏取りする。hook スクリプト自体の修正検証は該当コード片を bash で手動再現すれば同一セッション内で実物確認できる。また「〇〇（他プロジェクト）から issue が来てる」という指示は、まず自プロジェクトの issue 一覧に〇〇起票のものが無いか先に確認し、〇〇のリポジトリを見に行くのはその後にする。

**なぜ有効か**: heredoc 誤認を踏まえ実データ照合してから動くようにしたら、誤検知起因の不要な対応作業を避けられた。注入時点で OPEN 表示だった issue が実際は CLOSED 済みだった事例も再確認で検知できた。watermark 計算の修正も手動再現でその場検証できた。「dotclaude-ops から issue が来てる」を自リポジトリ側から先に見て、指示の実意図（自プロジェクトへ送られた issue）と一致した対応ができた。

**よくある失敗パターン**: hook の警告文・注入スナップショットを「機構からの情報だから正しい・最新」と無条件に信じて対応を始める。「〇〇から」を「〇〇のリポジトリを見る」と機械的に解釈して別リポジトリの issue 一覧を漁る。

## 15. 【PR提出は実マージ履歴で】

**何をするか**: upstream への PR は、ベースブランチ・rebase/cherry-pick の起点を fork 内の分岐関係でなく upstream 側の実際のマージ履歴（`git log --oneline upstream/<branch>`）で決める。CLAUDE.md の一般則（`git rebase --onto` 等）も無条件適用せず、対象ブランチのコミット内容を `git log --stat` で確認してから手段を選ぶ。PR 作成は `git push` が出力する fork 基準の URL を使い、base ブランチは明示的に切り替える。

**なぜ有効か**: Phase 分割 PR で後続の base が `master` でなく前 Phase の作業ブランチ（`vue3`）になること、作業ブランチにローカル専用コミットが混在し単純 rebase では PR にローカルパッチが混入することを、実データ確認で提出前に検知できた。upstream 側起点の PR 作成 URL が fork のブランチを解決できない問題も push 出力の URL 利用で回避できた。

**よくある失敗パターン**: fork 内のブランチ分岐関係と CLAUDE.md の一般則だけを根拠に PR の base・rebase 起点を決める。

## 16. 【コンテナ機構は個別に実確認】

**何をするか**: claude-container の組み込み機構（`GH_TOKEN_FILE` 等の自動 export・リビルド不要性）を独自追加の変数へ類推適用しない。`.claude-container.d/env` は既知変数のみをハードコード処理する設計で任意変数（`_FILE` 有無を問わず）を素通ししない。`GH_TOKEN_SECONDARY_FILE` は自動 export されず（bind mount のみ）`GH_TOKEN=$(cat <path>) gh ...` と都度明示指定する。`/workspace` は双方向マウントでコンテナ内の一時ファイル（`.env.local` 等）が即座にホストへ波及する。`env` コマンドが deny されている環境では shell 組み込みの `export` で代替する。いずれも README の変数表・実環境変数・ファイル実在で個別に確認する。

**なぜ有効か**: 組み込み `_FILE` 機構の挙動を独自変数（`VITE_MAPTILER_DEV_KEY`）へ類推適用して「渡せるはず」と誤認した手戻りを、実環境での個別確認で検知・修正できた。セカンダリトークンが `env` で空を返すのを「未設定」と誤診断した事例も、明示指定で実際にクロスリポジトリ投稿が成功し解消した。双方向マウントの把握でホスト側 dev サーバへの想定外の波及も防げた。

**よくある失敗パターン**: 組み込み機構の README 記載を独自変数にも当てはまる汎用保証と読み、実環境で確認せずユーザーに準備作業を依頼する。`env` でセカンダリ変数が空なのを未設定の根拠にする。

## 17. 【キー制限は実効範囲で評価】

**何をするか**: API キーの保護機構は、どの経路に実際に効くかを公式ドキュメントで確認してから設計する（MapTiler の user-agent 制限はブラウザアプリに機能せず、origin 制限は「他者の本番ドメインでの無断使用」を防ぐのみでローカル使用は防げない。形式もプロトコル・ポート抜きのドメイン名のみで `http://localhost:5173` は弾かれる）。アクセス不能リソースへの自動フォールバック（カスタムスタイル 404→標準スタイル）があると権限エラーが隠れることも把握しておく（グローバル原則11【権限は実効で設計する】のプロジェクト具体化）。

**なぜ有効か**: 開発用キーの防御の本質はキー値自体の非公開性で origin 制限は保険的位置づけ、と実効範囲を先に確定したことで、効かない制限への過信と「404＝致命的エラー」の誤診断（実際はフォールバックで地図表示は機能）の両方を避けられた。

**よくある失敗パターン**: 制限機構の名前から防御範囲を推測して設定し、実際に効く経路・フォールバック挙動を確認しない。

## 19. 【Vue 3の契約差分は実ソース・コンパイラ出力で確定】

**何をするか**: Vue 2→3 / Buefy 0.8→3.0 移行で同名・類似名の API を使うときは、グローバル原則14【API互換は契約全体で照合】のプロジェクト具体化として、(a) prop の実在（`<b-modal>` は `active` prop を持たず `modelValue` のみ。`:active` も `v-model:active` も機能せず `v-model="x"` だけが機能する）を `node_modules` 内のビルド前ソースへの grep で確定する。旧 prop は一律削除でなくコンポーネントごとに実物 props（ブラウザ上で mixin 再帰マージ込みに抽出）を確認する（`b-tooltip` は今も `active` を持つ）。(b) フレームワークのコンパイル規則（named slot と並ぶ v-slot なしの裸 `<template>` は inert な DOM 要素になる、`v-model` と明示 `@update:model-value` の併記は両方発火する、Vue3 コンパイラは `_resolveComponent` を `v-if` の外側へホイストして無条件実行するため到達不能なデッドコードでも `Failed to resolve component` 警告が新規に出る）を `@vue/compiler-dom` のコンパイル出力で確認する。(c) 属性の種類で変わる fallthrough（独自 prop はディレクティブに届かず、ネイティブイベントは内側の `<input>` までフォールスルーする）と、自前ラッパー層を書くときの移植元 teardown 契約（map 参照を undefined 化しない等）を移植元ソースと突き合わせる。フォールスルーした未知属性は DOM 上に素の camelCase 風属性（`current="1"` 等）として痕跡を残し、静的解析（実物 props×テンプレート AST 突合）と独立した検知経路になる。

**なぜ有効か**: 「Vue 2 で動いていた・prop 名として自然」な `:active` バインディングがリポジトリ内全10箇所で機能せず Phase 2 以降モーダルが一切開かないバグを抱えていたが、実ソース grep で存在 prop を確定して解消できた。裸 `<template>` の inert 化・二重発火・fallthrough 差・teardown 契約破壊・数週間持ち越した `svgicon` 未解決警告（upstream Vite 移行の消し忘れタグ）も、いずれもビルドは通りコンソールエラーも出にくい同型の罠として、コンパイラ出力・移植元ソースの照合で機械的に確定できた。

**よくある失敗パターン**: 「旧バージョンで動いていた構文だから」「prop 名として自然だから」と、実在確認せずにバインディング・スロット構文を書く。特定の属性名を機械的に全置換する。ラッパー層を書くとき移植元の teardown 契約を確認しない。原因不明のまま複数セッション持ち越す。

## 21. 【バグは型で全数照合】

**何をするか**: 1件のバグを見つけたら、そのバグの「型」（例: Buefy 3 で `modelValue` を新規に持つようになり旧 prop 名を失ったコンポーネント全て）を定義し、機械的に全数照合して同型バグを一掃する。単発修正で済ませない。フォールスルー系バグはビルドが通り console に警告も出ないため、console 監視でなく「操作→期待する状態変化」の behavior assertion で検知する。自動検証環境は移行の初期フェーズから用意する（同型バグを後段まで生かすコストは指数的に膨らむ）。

**なぜ有効か**: Phase 2 で入った `<b-modal>` の v-model 破損を1件直した際に型を横展開しなかったため、同型バグが他3種12箇所も放置され Phase 5 まで生き延びた。ユーザー指摘後に型を定義し全数照合して一掃できた。Phase 1〜4 で自動検証環境を用意しなかった代償が後段で大きく膨らんだと確認できた。

**よくある失敗パターン**: 1件のバグを個別修正で終わらせ、同じ型のバグが他所に無いか機械的に照合しない。ビルドが通り console エラーが出ないことを「正常」と読む。

## 22. 【devワークフローの破壊的操作は対象を確定】

**何をするか**: コンテナ dev ワークフローでの破壊的シェル操作は対象を先に確定してから実行する。(a) ファイル削除は `rm -f *.json` のようなワイルドカードを避け、個別指定するか `ls` で対象確認してから実行する（リポジトリルート直下では `package.json` 等の実ファイルを巻き込む）。(b) プロセス kill は `pkill -f <広いパターン>` を避け、`ps aux` で PID 確認後 `kill -9 <pid>` で個別指定する（広いパターンは Bash ツール自体の実行チェーンを巻き込み exit 144 で後続を未実行にする）。(c) ブランチ切替を伴う dev サーバー運用は、切替前に確実に vite/npm プロセスを kill してから `npm install`→再起動の順を徹底する（生きたプロセスが新旧 node_modules 混在で身に覚えのない ESLint/500 エラーを吐く）。(d) 依存追加・削除後の 504 や `ENOENT: <pkg>/dist/...` はコードでなく `optimizeDeps` キャッシュ陳腐化を疑い、`pkill vite`+`rm -rf node_modules/.vite`+再起動で切り分ける。

**なぜ有効か**: `rm -f /workspace/*.json` が `package.json`・`package-lock.json`・`jsconfig.json` を削除しかけた（git checkout で復元できたが一歩間違えば深刻な損失）。広い `pkill -f "node_modules/.bin/vite"` が複数回 Bash チェーンを巻き込んだ。プロセス停止順の徹底とキャッシュ掃除で、身に覚えのないエラーの切り分けができるようになった。

**よくある失敗パターン**: ワイルドカード rm・広い pkill で意図しない対象を巻き込む。ブランチ切替時に旧プロセスを止めずに npm install して混在エラーを出す。依存変更後の Vite エラーをコード側の問題と誤診断する。
