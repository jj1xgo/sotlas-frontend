# Best Practices

> `lessons.md` の 70 項目を蒸留した高レベル原則集。
> 詳細な教訓・事例は `lessons.md` を参照。
> グローバル層（`~/.claude/best_practices.md`）収載済みの原則は再掲せず、
> 本ファイルにはプロジェクト固有の具体化のみを置く。

## 1. 【fork状態は実データで】

**何をするか**: fork/upstream の同期・マージ・push 状態、ブランチ間の内容差（検証環境設定の有無・コミットの影響ファイル全量）は、patch-id 比較や handover の記載を鵜呑みにせず、実データ（`git diff A B -- <path>` の内容 diff、`git fetch` + 実リモートの `git log origin/<branch>`・`git ls-remote`、`git show --stat`）で確認する。ブランチ切替を伴う計画では、切替先に検証手段（`.mcp.json` 等）が実在するかも計画段階で確認する。

**なぜ有効か**: patch-id 比較の誤判定（1コミット複数目的）と、handover の「未push」記載が実リモート状態と食い違っていた事例（rebase 以前の旧系譜が残存）を実データ照合で検知できた。単一ブランチにのみ存在する機能コミットの一部ファイルだけをコピーして「動くはず」と誤認するリスクも、`git show --stat` での全ファイル確認で塞げた。切替先ブランチの MCP 設定欠如も計画段階のレビューで検知できた。

**よくある失敗パターン**: コミット単位のツール出力や引き継ぎ記載を鵜呑みにして同期・push状態を断定する。ブランチが違っても検証環境は同じだと決め打つ。

## 2. 【記述は実装・実データで裏取り】

**何をするか**: 「これはこの fork/プロジェクトだけのローカル事情」「一覧はこれで尽くしている」といったドキュメント記述は、upstream の履歴・PR（`git log --all`・`git branch --contains`）や `git ls-files`/`git log` 等の実データと定期的に突き合わせる。グローバル層とプロジェクト層の両方に同種の運用規約（署名ルール等）が書かれている場合は、記述同士の矛盾も定期的に突き合わせる。

**なぜ有効か**: 「ローカルパッチ」と思っていた FontAwesome 対応が実は upstream にマージ済み（PR #43）だったケースと、「master固有のみ」と判断した `.claude/` 一覧記述が実際には master にコミット済みの調査ノートを見落としていたケースの両方を、実データ照合で訂正できた。署名ルールの層間矛盾も定期レビュー（claude-md-panel）での突き合わせで発覚・解消できた。

**よくある失敗パターン**: CLAUDE.md 等の記述だけを根拠に現状（ローカル事情か否か・一覧の網羅性・規約の一貫性）を判断する。

## 3. 【動作検証は実行経路全体で】

**何をするか**: 「このライブラリは Vue 3 / Vite で動くか」の検証は、Node 単体の require/import・`renderToString` による SSR だけで完了とせず、`npm run build`・実ブラウザ（`app.mount()` まで）を通す。Node/SSR のモジュール解決やライフサイクル実行は実バンドラ・実クライアントと異なりうる。

**なぜ有効か**: Node 単体検証は CJS/ESM 解決の違いで `install()` 本体に到達しない偽陰性を生み、`renderToString` は `mounted()` 等を実行しないため子コンポーネントの自己登録設計を偽陰性のまま見逃した。実経路まで通す検証を完了条件にしてどちらも捕捉できるようになった。

**よくある失敗パターン**: Node/SSR でのスクリプトが例外を投げなかったことを「実ブラウザでも動く」と読む。

## 4. 【描画の暗黙デフォルトを明示】

**何をするか**: 描画ライブラリの省略時デフォルト（例: MapLibre の `fill` レイヤーは `fill-outline-color` 未指定だと `fill-color` と同色でアウトラインを描く）が絡む見た目のアーティファクトは、明示指定（透明化等）で潰す。半透明ポリゴンの重ね合わせでは特に注意する。

**なぜ有効か**: 縮退辺を持つポリゴンで自動アウトラインが重なり線状アーティファクトになった。明示的な透明化で解消し、以後の重ね描画実装の定石になった。

**よくある失敗パターン**: アーティファクトの原因を自分のジオメトリ生成側だけに探し、レンダラの暗黙仕様を疑わない。

## 5. 【許可リストは実走で確定】

**何をするか**: allowed-domains 等の環境許可リストは、`package.json` の依存を全種類（registry / `github:` / `git+https:` 等）洗い出して設計し、コンテナ内で `npm ci` が通り、対象機能（外部 API を叩くウィジェット等）を実際に動かすまで完成としない。

**なぜ有効か**: `github:` 形式の依存が `codeload.github.com` から tarball を取ることを、registry.npmjs.org だけ許可して実走するまで見落としていた。実行時にも Wikipedia ウィジェットの通信先（`en.wikipedia.org`）が未収載でエラートーストが出る漏れが見つかり、実行時通信まで実走確認の対象に含めるようになった。

**よくある失敗パターン**: 依存の取得経路を registry 一種と仮定し、ビルドが通った時点で許可リストを完成とみなす。

## 6. 【メジャーバージョン移行は契約差分まで確認】

**何をするか**: ライブラリのメジャーバージョン間互換性は、props/emits 等の名前一致だけで判断せず、呼び出し契約（スロットの渡し方・引数の必須化・属性フォールスルーの効き方・dispose/teardown 時に何を保持するかの暗黙契約）と、フレームワーク側のコンパイル規則（裸 `<template>` の inert 化、`v-model` と明示 `@update:model-value` の両発火等）を公式サンプル・コンパイラ出力・移植元ソースで照合する。

**なぜ有効か**: Buefy 0.8→3.0 で props/emits 名は不変だったがスロット呼び出し契約が変わっており、ビルドは通り見た目も部分的に出る気づきにくいバグを公式サンプルとの照合で検知できた。同名メソッドの引数必須化（`LoadingProgrammatic.open`）、属性の種類（独自 prop vs ネイティブイベント）で変わる fallthrough 挙動、自前ラッパー層が移植元の teardown 契約（map 参照を undefined 化しない等）を壊して子コンポーネントの TypeError を招いた事例も、同型のパターンとして契約照合で捕捉・解消できた。

**よくある失敗パターン**: props/emits 名やメソッド名が一致していることを互換性ありの証拠として扱う。ラッパー層を書くとき移植元の teardown 契約を確認しない。

## 7. 【git操作の副作用を先に把握】

**何をするか**: コミット直後の `git checkout` によるブランチ切替は「現在 HEAD で tracked だが切替先に存在しないファイル」を黙って削除する。また `git add` は複数パススペックのうち1つでも一致しないと全体を失敗させる。これらの副作用を踏まえ、切替直後は `git status` で状態を確認してから次の操作をする。

**なぜ有効か**: 別ブランチへのコミット直後に元ブランチへ切り替えた際、事前に計画していた手動 `rm` が不要（自動削除済み）だと `git status` 確認で気づけた。既に `git rm` 済みのパスを他の `git add` と混在させて失敗するミスも防げるようになった。

**よくある失敗パターン**: git 操作の副作用を知らずに、切替後・add の前後で想定通りの状態になっていると決め打ちする。

## 8. 【運用手順はskill化してtrigger明記】

**何をするか**: たまにしか使わない手順を CLAUDE.md の外（プロジェクト skill `.claude/skills/<name>/SKILL.md`）に出すときは、frontmatter の `description`（+`when_to_use`）に呼び出されるべき具体的な trigger 文言を書き込む。

**なぜ有効か**: 「CLAUDE.md の外に出すと呼ばれない限り読まれなくなる」という懸念は、skill の description が常時コンテキストの一覧に載り続ける仕組み（公式ドキュメント確認済み）により、trigger 文言の明記だけで実質的に解消できた。

**よくある失敗パターン**: 手順を skill へ切り出す際、description を一般的な要約のまま済ませ、呼び出されるべき具体的な状況を書かない。

## 9. 【型定義は読み、公開可否は実行で確認】

**何をするか**: 配布パッケージの公開型定義（`.d.ts`）とその docstring は、ライブラリ挙動や ToS/仕様グレーの評価の一次情報として先に読む。ただし `export declare` の存在を runtime export の証拠とせず、「公開 API か」は実際の import テストの実行まで裏取りする。

**なぜ有効か**: `@maptiler/sdk` のセッション課金機構が公開 doc コメントに明記されていると確認でき、「隠された内部実装のリバースエンジニアリング」という当初評価を「公言された動作の再現」へ覆せた。同時に `MAPTILER_SESSION_ID` は型定義上 `export declare` でも実際は未 export と import テストで確定できた。

**よくある失敗パターン**: 型定義の存在だけで「公開 API・利用可」と断定する。docstring を読まずに内部実装の推測でグレー評価を下す。

## 10. 【mapコントロールはload前参照で登録】

**何をするか**: MapLibre 系コントロール（mapbox-gl-draw 等）の `addControl` は、「map の `load` 後にしか解決しない reactive な値」経由でなく、ラッパーライブラリが提供する「map 生成直後（load 前）に解決済みの参照」（`@indoorequal/vue-maplibre-gl` の `mapSymbol` `ShallowRef` 等）経由で行う。

**なぜ有効か**: mapbox-gl-draw の `onAdd` は load 済み map に対して `map.on('load', ...)`（既発火済みなら二度と発火しない）＋16ms ポーリングへフォールバックする。load 前参照で登録するようにしたら、ポーリング頼みの不安定な初期化を避けられた。

**よくある失敗パターン**: 「map が手に入ってから addControl すれば安全」と考え、load 後解決の参照経由で初期化経路を不安定にする。

## 11. 【通信はtimeoutで打ち切る】

**何をするか**: UI のローディング状態を伴う HTTP リクエスト（axios 等）には `timeout` オプションを必ず設定する。`.catch()` だけでは応答が一切返らないネットワークハングを救えない。

**なぜ有効か**: `.then()` も `.catch()` も呼ばれず永久 pending になるケースがあると分かり、timeout を必須にしたらローディング解除漏れを設計段階で潰せるようになった。

**よくある失敗パターン**: 「エラーは `.catch()` で拾っているから大丈夫」と、無応答ハングのケースを考慮しない。

## 12. 【自動操作の観測はコード発火で裏取る】

**何をするか**: ブラウザ自動操作（Claude in Chrome 拡張等）はコンテナが自動操作ツールを持たない場合の代替検証手段になるが、合成イベントによる操作（地図ドラッグ等）が実ハンドラにコミットされないことがある。自動操作経由で不可解な挙動を観測したら、コード側該当パス（watch・イベントリスナー等）へのデバッグログで発火有無を確認し、未発火ならユーザーの実手動操作での再現有無を確認してから「バグ」か「テスト手法のアーティファクト」かを確定する。OS ネイティブダイアログ等は自動操作では確認できない制約も踏まえる。検証に必要な稀なデータ組み合わせは、ブラウザの目視探索でなくバックエンド API を直接クエリして機械的に絞り込む方が早い。

**なぜ有効か**: MapLibre の `dragPan` が合成マウスイベントでは内部カメラ状態を変えない現象を、切り分け（3ラウンド全てでログ発火なし＋手動操作で非再現）によりテスト手法側のアーティファクトと確定でき、無用なコード修正を避けられた。「写真なし・ルートあり」の稀な Summit ページも API 直接クエリで効率的に特定できた。

**よくある失敗パターン**: 自動操作ツール経由の観測をそのまま実ユーザー環境の挙動として扱い、バグ修正を始める。

## 13. 【hook警告・注入情報は実物で裏取る】

**何をするか**: コマンド文字列を正規表現で grep する設計の hook は、heredoc 内のリテラルテキストまで実行コマンドと誤認しうる。また session-start hook が自動注入する issue スナップショット等は取得時点のもので、その後の変化を反映しない。hook からの警告・注入情報は鵜呑みにせず、実データ（`git status`/`git log`・`gh issue view`）で裏取りしてから行動する。hook スクリプト自体の修正検証は、該当コード片を bash で手動再現すれば次回セッションを待たず同一セッション内で実物確認できる。

**なぜ有効か**: heredoc 誤認の特性を踏まえ警告を実データ照合してから動くようにしたら、誤検知起因の不要な対応作業を避けられるようになった。注入時点で OPEN 表示だった issue が実際は CLOSED 済みだった事例も実データ再確認で検知できた。watermark 計算の修正も手動再現でその場で検証できた（グローバル原則2【完了は実物で締める】の hook 入力版）。

**よくある失敗パターン**: hook の警告文・注入スナップショットを「機構からの情報だから正しい・最新」と無条件に信じて対応を始める。

## 14. 【issueの出所は自リポジトリから疑う】

**何をするか**: 「〇〇（他プロジェクト）から issue が来てる」という指示は、まず「このプロジェクト自身の issue 一覧に〇〇起票のものが無いか」を先に確認する。〇〇のリポジトリを見に行くのはその後。

**なぜ有効か**: 起票先が可変なプロジェクト間連携で「dotclaude-ops から」＝「dotclaude-ops リポジトリの issue 一覧」と早合点せず自リポジトリ側を先に見るようにしたら、指示の実際の意図（自プロジェクトへ送られた issue）と一致した対応ができた。

**よくある失敗パターン**: 「〇〇から」を「〇〇のリポジトリを見る」と機械的に解釈して別リポジトリの issue 一覧を漁る。

## 15. 【PR提出は実マージ履歴で】

**何をするか**: upstream への PR は、ベースブランチ・rebase/cherry-pick の起点を fork 内の分岐関係でなく upstream 側の実際のマージ履歴（`git log --oneline upstream/<branch>`）で決める。CLAUDE.md の一般則（`git rebase --onto` 等）も無条件適用せず、対象ブランチのコミット内容を `git log --stat` で確認してから手段を選ぶ。PR 作成は `git push` が出力する fork 基準の URL を使い、base ブランチは明示的に切り替える。

**なぜ有効か**: Phase 分割 PR で後続の base が `master` でなく前 Phase の作業ブランチ（`vue3`）になること、作業ブランチにローカル専用コミットが混在し単純 rebase では PR にローカルパッチが混入することを、実データ確認で提出前に検知できた。upstream 側起点の PR 作成 URL が fork のブランチを解決できない問題も push 出力の URL 利用で回避できた。

**よくある失敗パターン**: fork 内のブランチ分岐関係と CLAUDE.md の一般則だけを根拠に PR の base・rebase 起点を決める。

## 16. 【コンテナ機構は個別に実確認】

**何をするか**: claude-container の組み込み機構（`GH_TOKEN_FILE` 等の自動 export・リビルド不要性）を独自追加の変数へ類推適用しない。`.claude-container.d/env` は既知変数のみをハードコード処理する設計で任意変数を素通ししない。`/workspace` は双方向マウントでコンテナ内の一時ファイル（`.env.local` 等）が即座にホストへ波及する。`env` コマンドが deny されている環境では shell 組み込みの `export` で代替する。いずれも README の変数表・実環境変数・ファイル実在で個別に確認する。

**なぜ有効か**: 組み込み `_FILE` 機構の挙動を独自変数（`VITE_MAPTILER_DEV_KEY`）へ類推適用して「渡せるはず」と誤認した手戻りを、実環境での個別確認で検知・修正できた。双方向マウントの把握でホスト側 dev サーバへの想定外の波及も防げるようになった。

**よくある失敗パターン**: 組み込み機構の README 記載を独自変数にも当てはまる汎用保証と読み、実環境で確認せずユーザーに準備作業を依頼する。

## 17. 【キー制限は実効範囲で評価】

**何をするか**: API キーの保護機構は、どの経路に実際に効くかを公式ドキュメントで確認してから設計する（MapTiler の user-agent 制限はブラウザアプリに機能せず、origin 制限は「他者の本番ドメインでの無断使用」を防ぐのみでローカル使用は防げない。形式もプロトコル・ポート抜きのドメイン名のみ）。アクセス不能リソースへの自動フォールバック（カスタムスタイル 404→標準スタイル）があると権限エラーが隠れることも把握しておく（グローバル原則11【権限は実効で設計する】のプロジェクト具体化）。

**なぜ有効か**: 開発用キーの防御の本質はキー値自体の非公開性で origin 制限は保険的位置づけ、と実効範囲を先に確定したことで、効かない制限への過信と「404＝致命的エラー」の誤診断（実際はフォールバックで地図表示は機能）の両方を避けられた。

**よくある失敗パターン**: 制限機構の名前から防御範囲を推測して設定し、実際に効く経路・フォールバック挙動を確認しない。

## 18. 【低レベル異常はキャッシュ・権限から確認】

**何をするか**: chromium の SIGTRAP クラッシュや `optimizeDeps` キャッシュ陳腐化による Vite dev server の 504/`ENOENT: <pkg>/dist/...` など、低レベルなクラッシュ・不可解なビルド/実行エラーに遭遇したら、グローバル原則17【異常はまず基本層を疑う】に従い、サンドボックス制約やコード不具合を疑う前に `ls -ld` での所有者確認、`pkill vite` + `rm -rf node_modules/.vite` + 再起動でのキャッシュ切り分けを先に行う。

**なぜ有効か**: chromium の SIGTRAP クラッシュは Podman の capability/seccomp でなく `~/.config` の root 所有（crashpad の DB パス解決失敗）が真因で、`XDG_CONFIG_HOME` の変更だけで解消した。npm 依存の追加・削除後の Vite キャッシュ陳腐化も同型の切り分けパターンとして機能した。

**よくある失敗パターン**: 低レベルシグナルや不可解なビルドエラーを見て真っ先にサンドボックス・コード側を疑い、権限・キャッシュの確認を後回しにする。
